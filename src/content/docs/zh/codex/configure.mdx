这页只讲一件事：**把 Codex 的请求打到 su8 codes**。

## 你需要的两个值

- Base URL：`https://su8.codes/v1`
- API Key：你在 su8 codes 后台生成的 key

## 最通用的鉴权写法

请求头：

```bash
Authorization: Bearer <你的API_KEY>
```

## Codex CLI（手动配置，最稳）

如果你用的是 Codex CLI，通常可以在 `~/.codex/` 配置“模型提供方（provider）”。

### 1) 打开配置目录

<Tabs>
  <TabItem label="Windows">
    先创建目录（如果还没有）：

    ```bat
    mkdir "%USERPROFILE%\\.codex"
    ```

    打开目录：

    ```bat
    start "" "%USERPROFILE%\\.codex"
    ```
  </TabItem>

  <TabItem label="macOS">
    ```bash
    mkdir -p "$HOME/.codex"
    open "$HOME/.codex"
    ```
  </TabItem>

  <TabItem label="Linux">
    ```bash
    mkdir -p "$HOME/.codex"
    xdg-open "$HOME/.codex" 2>/dev/null || ls -la "$HOME/.codex"
    ```
  </TabItem>
</Tabs>

### 2) 创建/修改两个文件

在该目录下准备：
- `config.toml`（告诉 Codex：请求往哪发，用哪个接口）
- `auth.json`（放你的 API Key）

`config.toml` 示例（你可以先照抄，跑通后再改模型）：

```toml
model_provider = "su8"
model = "gpt-5.2"
network_access = "enabled"
disable_response_storage = true

[model_providers.su8]
name = "su8"
base_url = "https://su8.codes/v1"
wire_api = "responses"
requires_openai_auth = true
```

`auth.json` 示例：

```json
{
  "OPENAI_API_KEY": "<你的API_KEY>"
}
```

> 这里用 `OPENAI_API_KEY` 这个名字，是为了兼容很多工具的“习惯叫法”。key 本质还是你的 su8 codes key。

### 3) 验证配置是否生效

去下一页跑一条请求就知道了：
- [跑通一次请求](/zh/codex/verify/)

## 如果你找不到这些配置项

不同版本的 Codex 配置方式可能不一样，但你要找的关键词通常是：
- `base_url`
- `api_key` / `OPENAI_API_KEY`

实在不行，先用最通用的 HTTP 调用跑通（curl / 代码里 fetch），再回头折腾 CLI：
- [跑通一次请求](/zh/codex/verify/)
